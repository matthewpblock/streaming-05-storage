## Custom Project

### Dataset

The weather producer in this repository does not read a static sales file; it
fetches a live 10-day hourly forecast for Honolulu from the Open-Meteo API at
runtime. The consumer writes its output to a local dataset file named
`data/output/honolulu_weather.csv` (and also persists records into
`data/output/weather.duckdb`). Each record represents an hourly forecast and
contains these fields:

- `location`: human-readable location string (e.g. "Honolulu, HI")
- `pulled_timestamp`: ISO timestamp when the forecast was pulled from the API
- `predicting_timestamp`: ISO timestamp the forecast applies to (the forecasted hour)
- `temperature_f`: temperature in degrees Fahrenheit (string-typed in the message)
- `humidity_pct`: relative humidity as an integer percent (string-typed in the message)
- `wind_speed_mph`: wind speed in miles per hour (string-typed in the message)
- `wind_direction_deg`: wind direction in degrees (string-typed in the message)

The producer does not use the original `data/sales.csv` dataset; the weather
stream is derived from the live forecast endpoint and the repository's
`honolulu_weather.csv` is a consumer-produced output file rather than an input.

### Kafka Messages

The Kafka producer sends one JSON-like dictionary per forecast hour. Messages
include the fields listed above and are sent to the topic configured by the
environment variable `KAFKA_TOPIC_WEATHER` (default: `streaming-05-weather`).
The producer sets a fixed message key of `Honolulu` so all Honolulu records
are partitioned consistently. Message fields are not otherwise transformed by
the producer — validation and enrichment happen on the consumer side.

### Consumer Processing

The consumer subscribes to the same topic (`streaming-05-weather` by default)
and consumes up to `MAX_MESSAGES` (configured to capture the full forecast).
For each message it:

- validates the required fields (`location`, `pulled_timestamp`,
 `predicting_timestamp`, `temperature_f`, `humidity_pct`, `wind_speed_mph`,
 `wind_direction_deg`)
- enriches the record by adding `temp_c` (temperature converted to Celsius)
 and `is_comfortable` (boolean flag based on temperature and humidity
 thresholds)
- writes the enriched record into a DuckDB database at
 `data/output/weather.duckdb` and appends a CSV row into
 `data/output/honolulu_weather.csv` using the canonical consumed field order
- logs a brief summary line for each processed forecast showing
 pulled/predicting timestamps, Celsius temperature, and the comfort flag

The consumer records Kafka metadata fields (such as `_kafka_key`,
`_kafka_partition`, `_kafka_offset`) as part of the consumed field list
written to CSV/DB.

### Experiments

- Phase 4 change: increased the consumer `MAX_MESSAGES` and tuned
 `MESSAGE_INTERVAL_SECONDS` in the producer to reliably capture the full
 10-day (240-hour) forecast without overwhelming the broker; this ensures the
 consumer can process the entire forecast in one run.
- Phase 5 application: added runtime enrichment and storage to enable a
 downstream analytics workflow — the consumer now computes `temp_c` and
 `is_comfortable`, persists enriched records to `weather.duckdb`, and outputs
 `honolulu_weather.csv` for easy BI/visualization. These persisted artifacts
 make it straightforward to run ad-hoc SQL queries or charts against the
 consumed stream.

### Results

When the producer and consumer are run together the producer emits the 10-day
hourly forecast (240 messages) and the consumer processes each valid
forecast, writing rows into `data/output/honolulu_weather.csv` and
`data/output/weather.duckdb`. The consumer logs a processed-summary line per
message and prints a final storage summary when it finishes; rejected or
malformed records (if any) are listed in warnings.

### Interpretation

- What changed from the original example: instead of streaming from a static
 sales CSV, this project streams a live weather forecast from an external API
 and demonstrates a real-time producer that supplies time-series forecasts.
- What we learned: watching messages move through Kafka highlights how message
 keys, topic configuration, and consumer read limits affect ordering,
 partitioning, and throughput. The end-to-end flow clarifies where validation
 should occur (consumer-side here) and where enrichment is most efficient.
- Business value: the stream can show near-term forecast trends (temperature,
 humidity, wind) hour-by-hour, enabling operational decisions such as
 staffing, event scheduling, or logistics adjustments for the Honolulu area.
- Business intelligence: the enriched, persisted stream supports simple BI use
 cases — counting "comfortable" hours, computing temperature trends, or
 joining weather forecasts with other datasets (sales, events) to quantify
 weather impact on operations.
